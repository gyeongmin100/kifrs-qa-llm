# Accounting-Standards-QA-LLM: 회계기준 특화 LLM (LoRA + RAG)

한국회계기준원과 금융감독원의 **회계기준 질의회신 사례**를 활용해, 오픈소스 LLM에 회계 도메인 지식을 적용하는 과정을 실험한 프로젝트입니다.

이 프로젝트에서는 다음 네 가지 구성을 같은 홀드아웃 문항으로 비교했습니다.

- **Base**: Qwen2.5-7B-Instruct 기본 모델입니다.
- **RAG**: 과거 유사 질의회신 사례를 검색해 근거로 제공합니다.
- **LoRA**: 회계기준과 문단을 인용하며 결론을 제시하는 답변 스타일을 파인튜닝합니다.
- **LoRA+RAG**: 파인튜닝한 모델에 검색 근거를 함께 제공합니다.

회계 도메인은 데이터 민감성이 높아 외부 API에 원문 데이터를 보내기 어려운 경우가 많습니다. 따라서 답변 생성 모델은 로컬/온프레미스 실행이 가능한 오픈소스 모델을 기준으로 구성했습니다. OpenAI API는 최종 평가 단계에서 LLM judge로만 사용했습니다.

## 아키텍처

```
KIFRS.com 질의회신 (2,255건)
        │  scrape.py
        ▼
원본 JSONL ── prepare_data.py ──► train.jsonl (2,114건)
                                  test.jsonl  (30건, 홀드아웃)
                                  rag_corpus.jsonl (train과 동일, test 제외)
        │
        ├─► [LoRA] Qwen2.5-7B-Instruct + bf16 LoRA (TRL SFTTrainer)
        └─► [RAG]  bge-m3 임베딩 + FAISS 유사 사례 top-3 검색
        │
        ▼
평가: 30문항 × 4구성, 임베딩 유사도 + gpt-5.4 LLM judge
```

## 프로젝트 진행 과정

### 1. 데이터 수집: 화면 스크래핑 대신 내부 API를 호출했습니다

처음에는 KIFRS.com 화면의 HTML을 파싱하려고 했습니다. 하지만 실제 사이트는 React 기반 SPA라서 HTML 안에 질의회신 본문이 들어 있지 않았습니다.

그래서 브라우저 개발자도구의 Network 탭을 확인해 화면이 호출하는 내부 JSON API를 찾았습니다. 이후 `src/scrape.py`에서 해당 내부 API를 직접 호출해 전체 2,255건을 JSONL로 저장했습니다.

이 방식은 브라우저 자동화보다 단순하고 안정적이었습니다. 화면 렌더링이나 HTML 구조에 의존하지 않고, 서버가 내려주는 구조화된 JSON을 그대로 수집할 수 있었기 때문입니다.

### 2. 데이터 전처리: 질의와 회신을 분리했습니다

수집한 원문은 질문과 답변이 하나의 `fullContent` 텍스트에 섞여 있었습니다. 출처별로 `질의/회신`, `질의 요약/회신 요약`, `Ⅲ. 회신`, `회신□...`처럼 표기가 달라서 단일 규칙으로는 안정적으로 분리하기 어려웠습니다.

그래서 `src/prepare_data.py`에서 정규식 기반 파서를 만들고, 실패 사례를 확인하면서 마커 패턴을 보강했습니다. 최종적으로 2,255건 중 2,144건(95.1%)을 질의-회신 쌍으로 파싱했습니다.

파싱된 2,144건 중 30건은 홀드아웃 테스트셋으로 분리했습니다. 이 테스트 문항은 학습 데이터뿐 아니라 RAG 인덱스에서도 제외해 데이터 누출을 막았습니다.

### 3. 모델 구성: Base, RAG, LoRA, LoRA+RAG를 비교했습니다

베이스 모델은 `Qwen2.5-7B-Instruct`를 사용했습니다. 온프레미스 실행 가능성을 고려해 7B급 오픈소스 모델을 선택했습니다.

회계 질의회신 특화를 위해 두 가지 접근을 비교했습니다.

- **LoRA**: 2,114건의 학습 데이터로 회계기준을 인용하며 결론을 쓰는 답변 스타일을 학습했습니다.
- **RAG**: bge-m3 임베딩과 FAISS로 과거 유사 질의회신 top-3를 검색해 프롬프트에 근거로 제공했습니다.

실험은 `base`, `rag`, `lora`, `lora_rag` 4구성으로 나누었습니다. 이렇게 구성해야 성능 변화가 파인튜닝 때문인지, 검색 근거 때문인지, 둘의 조합 때문인지 분리해서 볼 수 있습니다.

### 4. 평가: 임베딩 유사도만으로는 부족했습니다

먼저 30개 홀드아웃 문항에 대해 4구성 답변을 생성하고, 실제 회신과의 bge-m3 임베딩 코사인 유사도를 계산했습니다. 이 평가는 전체 경향을 빠르게 볼 수 있지만, “그럴듯하게 비슷한 문장”과 “실제로 맞는 결론”을 구분하지 못하는 한계가 있었습니다.

그 한계를 보완하기 위해 `src/llm_judge.py`를 만들었습니다. OpenAI `gpt-5.4`를 judge로 사용해 질의, 정답, 모델 답변을 함께 제공하고, 문체나 분량이 아니라 최종 결론이 일치하는지만 O/X로 판정하도록 했습니다.

## 실험 결과

홀드아웃 30문항에 대해 4가지 구성으로 답변을 생성하고, 실제 회신과의 bge-m3 임베딩 코사인 유사도를 측정했습니다. 실험 조건은 Qwen2.5-7B-Instruct, LoRA r=16, 2 epochs, A100 80GB입니다.

| 구성 | 정답 유사도(평균) | 표준편차 |
|---|---:|---:|
| base | 0.756 | 0.074 |
| **rag** | **0.774** | 0.059 |
| lora | 0.721 | 0.094 |
| lora+rag | 0.731 | 0.085 |

### 임베딩 유사도 해석

RAG는 평균 유사도 기준으로 base보다 높은 점수를 보였습니다. 유사 사례를 근거로 제공하는 방식이 답변의 표면적 유사도에는 도움이 된 것으로 볼 수 있습니다.

다만 LoRA 구성에서는 반복 생성 문제가 나타났습니다. 답변 원문을 검토한 결과, 동일 문장을 반복하는 붕괴 현상이 여러 문항에서 발견되었습니다. 이는 greedy decoding과 파인튜닝 모델 조합에서 발생할 수 있는 실패 모드로 보이며, `repetition_penalty` 적용 후 재평가가 필요합니다.

또한 임베딩 유사도만으로는 결론의 정오를 판단하기 어렵다는 점도 확인했습니다. 예를 들어 한 문항에서는 base 모델이 정답과 정반대 결론을 내고 중국어 환각까지 출력했지만, 유사도는 0.80 수준으로 높게 나왔습니다. 따라서 임베딩 유사도는 참고 지표로만 사용하고, 결론 일치 여부를 별도로 평가해야 한다고 판단했습니다.

### 결론(O/X) 기반 정밀 재평가

임베딩 유사도의 한계를 보완하기 위해 OpenAI `gpt-5.4`를 judge로 사용했습니다. 문체와 서술 방식은 무시하고, 정답과 모델 답변의 최종 결론이 일치하는지만 판정했습니다.

| 구성 | 결론 정답률 |
|---|---:|
| base | 6.7% (2/30) |
| rag | 10.0% (3/30) |
| lora | 20.0% (6/30) |
| **lora+rag** | **26.7% (8/30)** |

gpt-5.4 judge 기준에서는 `lora_rag`가 가장 높았습니다. 하지만 절대 정답률은 여전히 낮았습니다. 임베딩 유사도에서 보였던 “RAG가 base보다 낫다”는 신호는 결론 일치 기준에서는 약했고, LoRA+RAG 조합만 상대적으로 나은 결과를 보였습니다.

문항 단위로 RAG/LoRA 추가가 정답을 뒤집은 방향을 보면 다음과 같습니다.

| 비교 | 오답→정답 | 정답→오답 |
|---|---:|---:|
| base → rag | 2건 | 1건 |
| lora → lora_rag | 6건 | 4건 |

RAG와 LoRA+RAG가 일부 문항을 정답으로 뒤집었지만, 표본이 30문항으로 작고 전반적인 정답률이 낮았습니다. 또한 30문항 중 17건(57%)은 4구성 전부 오답이었고, 4구성 전부 정답인 문항은 1건뿐이었습니다.

## 결론

이 프로젝트에서 가장 중요한 발견은 두 평가 방식이 서로 다른 그림을 보여준다는 점입니다.

임베딩 유사도는 “정답과 비슷하게 들리는 답변”에 점수를 줍니다. 반면 결론 O/X 평가는 “실제로 맞는 결론을 냈는가”를 봅니다. 회계, 법률, 의료처럼 결론의 정확성이 중요한 도메인에서는 임베딩 유사도만으로 모델을 평가하면 위험합니다.

이번 실험에서는 LoRA+RAG가 결론 정답률 기준으로 가장 높았지만, 실무적으로 충분한 수준에는 도달하지 못했습니다. 7B급 모델에 검색과 경량 파인튜닝을 적용하는 것만으로는 갑설/을설이 대립하는 복합 회계판단 문항을 안정적으로 해결하기 어렵다는 점을 확인했습니다.
