# KIFRS-QA-LLM: 회계기준 질의회신 특화 LLM (LoRA + RAG)

한국회계기준원·금융감독원의 **회계기준 질의회신 사례**를 활용해, 오픈소스 LLM에 도메인 지식을 주입하는 두 가지 접근을 구현하고 비교한 프로젝트입니다.

- **LoRA 파인튜닝** — 회계기준·문단을 인용하며 결론을 제시하는 회계 자문 답변 스타일 학습
- **RAG** — 과거 유사 질의회신 사례를 검색해 근거로 제공
- **비교 평가** — Base / RAG / LoRA / LoRA+RAG 4개 구성을 동일한 홀드아웃 문항으로 평가

회계 도메인은 데이터 민감성이 높아 외부 API 대신 **로컬(온프레미스) LLM**이 요구되는 경우가 많습니다. 이 프로젝트는 오픈소스 모델(Qwen2.5-7B-Instruct)만으로 도메인 특화 어시스턴트를 구축하는 전 과정을 다룹니다.

## 아키텍처

```
KIFRS.com 질의회신 (2,255건)
        │  scrape.py
        ▼
원본 JSONL ── prepare_data.py ──► train.jsonl (2,114건)
                                  test.jsonl  (30건, 홀드아웃)
                                  rag_corpus.jsonl (train과 동일, test 제외)
        │
        ├─► [LoRA] Qwen2.5-7B-Instruct + bf16 LoRA (TRL SFTTrainer)
        └─► [RAG]  bge-m3 임베딩 + FAISS 유사 사례 top-3 검색
        │
        ▼
평가: 30문항 × 4구성, 정답(실제 회신) 대비 임베딩 유사도 + 정성 비교
```

## 프로젝트 진행 과정

### 1. 데이터 수집: 화면 스크래핑 대신 내부 API 호출

처음에는 KIFRS.com 화면의 HTML을 파싱하려고 했지만, 실제 사이트는 React 기반 SPA라 HTML 안에 질의회신 본문이 들어 있지 않았다. 브라우저 개발자도구의 Network 탭을 확인해 화면이 호출하는 내부 JSON API를 찾았고, `src/scrape.py`에서 해당 API를 직접 호출해 전체 2,255건을 JSONL로 저장했다.

이 방식은 브라우저 자동화보다 단순하고 안정적이었다. HTML 구조나 화면 렌더링에 의존하지 않고, 서버가 내려주는 구조화된 JSON을 그대로 수집할 수 있었기 때문이다.

### 2. 데이터 전처리: 질의와 회신 분리

수집한 원문은 질문과 답변이 하나의 `fullContent` 텍스트에 섞여 있었다. 출처별로 `질의/회신`, `질의 요약/회신 요약`, `Ⅲ. 회신`, `회신□...`처럼 표기가 달라서 단일 규칙으로는 분리하기 어려웠다.

그래서 `src/prepare_data.py`에서 정규식 기반 파서를 만들고, 실패 사례를 확인하면서 마커 패턴을 보강했다. 최종적으로 2,255건 중 2,144건(95.1%)을 질의-회신 쌍으로 파싱했고, 이 중 30건은 홀드아웃 테스트셋으로 분리했다. 테스트 문항은 학습 데이터뿐 아니라 RAG 인덱스에서도 제외해 데이터 누출을 막았다.

### 3. 모델 구성: Base, RAG, LoRA, LoRA+RAG 비교

베이스 모델은 `Qwen2.5-7B-Instruct`를 사용했다. 온프레미스 실행 가능성을 고려해 7B급 오픈소스 모델을 선택했고, 회계 질의회신 특화에는 두 가지 접근을 비교했다.

- LoRA: 2,114건의 학습 데이터로 회계기준을 인용하며 결론을 쓰는 답변 스타일을 학습
- RAG: bge-m3 임베딩과 FAISS로 과거 유사 질의회신 top-3를 검색해 프롬프트에 근거로 제공

실험은 `base`, `rag`, `lora`, `lora_rag` 4구성으로 나눴다. 이렇게 해야 성능 변화가 파인튜닝 때문인지, 검색 근거 때문인지, 둘의 조합 때문인지 분리해서 볼 수 있다.

### 4. 평가: 임베딩 유사도만으로는 부족했다

먼저 30개 홀드아웃 문항에 대해 4구성 답변을 생성하고, 실제 회신과의 bge-m3 임베딩 코사인 유사도를 계산했다. 이 평가는 빠르게 전체 경향을 볼 수 있지만, “그럴듯하게 비슷한 문장”과 “실제로 맞는 결론”을 구분하지 못하는 한계가 있었다.

그 한계를 보완하기 위해 `src/llm_judge.py`를 만들어 OpenAI `gpt-5.4`를 judge로 사용했다. judge에는 질의, 정답, 모델 답변을 함께 주고 문체나 분량이 아니라 최종 결론이 일치하는지만 O/X로 판정하게 했다.

## 실험 결과

홀드아웃 30문항에 대해 4가지 구성으로 답변을 생성하고, 실제 회신(정답)과의 bge-m3 임베딩 코사인 유사도를 측정했다. (Qwen2.5-7B-Instruct, LoRA r=16, 2 epochs, A100 80GB)

| 구성 | 정답 유사도(평균) | 표준편차 |
|---|---|---|
| base | 0.756 | 0.074 |
| **rag** | **0.774** | 0.059 |
| lora | 0.721 | 0.094 |
| lora+rag | 0.731 | 0.085 |

### 해석: 숫자 그대로 읽으면 안 되는 이유

**RAG는 일관된 개선**을 보였다(+0.018, 표준편차도 감소). 유사 사례를 근거로 제공하는 방식이 사실 기반 답변에 유효함을 확인했다.

**LoRA 점수 하락은 모델 품질이 아니라 디코딩 문제**로 확인됐다. 답변 원문을 검토한 결과 LoRA 모델에서 동일 문장을 반복 생성하는 붕괴 현상(greedy decoding + 파인튜닝 조합의 알려진 실패 모드)이 다수 발견됐고, 이것이 점수를 끌어내렸다. `repetition_penalty` 적용 후 재평가가 다음 개선 과제다.

**자동 지표의 한계도 확인했다.** 임베딩 유사도는 "주제가 비슷하고 길게 쓴 오답"에 관대하다:

- 한 문항(기타포괄손익-공정가치 지분상품 처분)에서 base 모델은 정답과 **정반대 결론**("재분류 가능" vs 정답 "재분류 불가")에 중국어 환각까지 출력했지만 유사도 0.80으로 고득점
- 반면 lora+rag는 다른 문항에서 정답 결론을 정확히 맞추고도 답변이 간결하다는 이유로 낮은 점수

이 때문에 임베딩 유사도는 참고 지표로만 쓰고, **결론 일치 여부(O/X) 기반 정성 평가를 병행**하는 것이 타당하다는 결론을 얻었다. 문항별 상세 점수는 `results/eval_similarity.csv`, 전체 답변 원문은 `results/eval_answers.json`에 있다.

### 결론(O/X) 기반 정밀 재평가: 다른 그림이 나왔다

임베딩 유사도의 한계를 보완하기 위해 OpenAI `gpt-5.4`를 judge로 사용해 "정답과 모델 답변의 최종 결론이 일치하는가"만 엄격히 판정했다(`src/llm_judge.py`, 근거는 `results/judge_scores_gpt-5.4.json`). 문체·서술 방식은 무시하고 결론(허용/불허, 자산/부채 분류 등)만 본다.

| 구성 | 결론 정답률 |
|---|---|
| base | 6.7% (2/30) |
| rag | 10.0% (3/30) |
| lora | 20.0% (6/30) |
| **lora+rag** | **26.7% (8/30)** |

**gpt-5.4 judge 기준에서는 lora+rag가 가장 높았지만, 절대 정답률은 여전히 낮다.** 임베딩 유사도에서 보였던 "RAG가 base보다 낫다"는 신호는 결론 일치 기준에서는 약했고, LoRA+RAG 조합만 상대적으로 나은 결과를 보였다.

문항 단위로 RAG/LoRA 추가가 정답을 뒤집은 방향을 추적하면:

| 비교 | 오답→정답 | 정답→오답 |
|---|---|---|
| base → rag | 2건 | 1건 |
| lora → lora_rag | 6건 | 4건 |

RAG와 LoRA+RAG가 일부 문항을 정답으로 뒤집었지만, 표본이 30문항으로 작고 전반적인 정답률이 낮다. 또한 30문항 중 17건(57%)은 4구성 전부 오답, 전부 정답인 문항은 1건뿐이었다. 즉 base가 못 푸는 문제는 RAG를 붙이든 파인튜닝을 하든 대부분 여전히 못 풀었다.

**해석**: 두 평가 방식이 서로 다른 결론을 낸 것 자체가 핵심 발견이다. 임베딩 유사도는 "그럴듯한 답변"에 점수를 주고, 결론 O/X는 "실제로 맞았는가"를 본다. 후자 기준으로는 LoRA+RAG가 상대적으로 가장 낫지만, 이 프로젝트의 데이터 규모(2,114건)·모델 크기(7B)·검색 범위(top-3)에서 실무적으로 충분한 정답률은 확보하지 못했다. 갑설/을설이 대립하는 복합 회계판단 문항은 7B급 모델이 검색·경량 파인튜닝만으로는 근본적으로 풀기 어려운 난이도일 가능성이 있다.

### 향후 개선
1. `repetition_penalty=1.15` 적용 후 재평가 (LoRA 반복 붕괴 해소)
2. 테스트 문항 확대(30→100+)로 통계적 유의성 확보
3. RAG top-k 확대, 리랭커 도입으로 검색 품질 개선
4. 학습률 조정(1e-4 → 5e-5) 재학습 실험
5. 문항 난이도별 층화 분석 (단순 사실 확인 vs 갑설/을설 대립형 판단 문항 분리 평가)

## 평가 설계

- **홀드아웃**: 테스트 30문항은 학습 데이터와 RAG 인덱스 양쪽에서 제외 (데이터 누출 방지)
- **자동 지표**: 모델 답변과 실제 회신(정답)의 bge-m3 임베딩 코사인 유사도
- **정성 평가**: 전체 답변을 `results/eval_answers.json`에 저장해 결론 일치 여부·기준서 인용 정확성·환각 여부를 수동 확인

## 데이터 파싱

원문은 출처(회계기준원 정규질의/신속처리질의, 금감원, IFRS 해석위원회)마다 섹션 표기가 달라(`질의/회신`, `질의 요약/회신 요약`, `Ⅲ. 회신`, `회신□...` 등) 정규식 기반 파서로 질문부/답변부를 분리했습니다. 전체 2,255건 중 2,144건(95.1%) 파싱에 성공했으며, 나머지는 회신 섹션이 없거나 비정형 구조인 문서입니다.
